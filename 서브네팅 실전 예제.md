# [Network] 서브네팅(Subnetting) 완벽 이해 및 실전 예제

## 1. 서브네팅(Subnetting)이란?
서브네팅은 **커다란 하나의 네트워크를 여러 개의 작은 네트워크(Subnet)로 쪼개는 작업**입니다.

* **왜 할까요?** 1. **IP 주소 낭비 방지:** 필요한 만큼만 IP를 할당하여 효율적으로 사용.
  2. **브로드캐스트 도메인 축소:** 한 네트워크 안에 너무 많은 PC가 있으면 불필요한 통신(브로드캐스트)으로 네트워크가 느려집니다. 이를 쪼개어 성능을 높이고 보안을 강화합니다.

## 2. 서브네팅 실전 예제 1: 반으로 쪼개기 (/25)

가장 흔하게 보는 네트워크 주소가 `192.168.0.0/24` 입니다. 
여기서 `/24` (서브넷 마스크: 255.255.255.0)는 전체 32비트 중 앞의 24비트가 '네트워크 부분', 뒤의 8비트가 '호스트(PC) 부분'이라는 뜻입니다. 즉, 2^8 = 256개의 IP를 쓸 수 있죠.

이 네트워크를 **2개의 부서(영업부, 개발부)가 나눠 쓰도록 서브네팅** 해보겠습니다.
2개로 나누려면 호스트 비트에서 **1비트**를 빌려와야 합니다. (2^1 = 2)

* **기존:** `/24` (호스트 비트 8개)
* **변경:** `/25` (호스트 비트 7개로 감소, 서브넷 마스크 `255.255.255.128`)

이제 할당 가능한 IP는 부서당 2^7 = 128개가 됩니다.

### [결과: 2개의 서브넷]
**① 1번 서브넷 (영업부)**
* **네트워크 ID:** `192.168.0.0` (가장 첫 IP, 대역을 나타냄)
* **사용 가능한 IP:** `192.168.0.1` ~ `192.168.0.126` (126개)
* **브로드캐스트 IP:** `192.168.0.127` (가장 마지막 IP, 전체 방송용)

**② 2번 서브넷 (개발부)**
* **네트워크 ID:** `192.168.0.128` 
* **사용 가능한 IP:** `192.168.0.129` ~ `192.168.0.254` (126개)
* **브로드캐스트 IP:** `192.168.0.255`

> 💡 **중요 공식:** 하나의 서브넷에서 실제 PC에 할당 가능한 IP 개수는 항상 **(전체 IP 개수) - 2** 입니다. 가장 첫 주소(네트워크 ID)와 가장 마지막 주소(브로드캐스트)는 기기에 할당할 수 없기 때문입니다.

## 3. 서브네팅 실전 예제 2: 4개로 쪼개기 (/26)

이번에는 `192.168.1.0/24` 네트워크를 **4개의 작은 네트워크로 쪼개보겠습니다.** (예: Safe-Path 프로젝트의 101동, 102동, 103동, 104동 센서망 분리)

4개로 쪼개려면 호스트 비트에서 **2비트**를 빌려와야 합니다. (2^2 = 4)

* **기존:** `/24`
* **변경:** `/26` (서브넷 마스크 `255.255.255.192`)
* **각 서브넷당 IP 개수:** 2^6 = 64개 (사용 가능 IP는 64 - 2 = 62개)

### [결과: 4개의 서브넷 대역]
1. **101동 대역:** `192.168.1.0` ~ `192.168.1.63` (사용 가능: `.1` ~ `.62`)
2. **102동 대역:** `192.168.1.64` ~ `192.168.1.127` (사용 가능: `.65` ~ `.126`)
3. **103동 대역:** `192.168.1.128` ~ `192.168.1.191` (사용 가능: `.129` ~ `.190`)
4. **104동 대역:** `192.168.1.192` ~ `192.168.1.255` (사용 가능: `.193` ~ `.254`)

## 4. 실전 Q&A

**Q. 클라우드(AWS)에서 서브넷을 나눌 때, Public Subnet과 Private Subnet은 어떻게 구성하시겠습니까?**

**A.** "네, 만약 VPC에 `10.0.0.0/16` 대역이 할당되어 있다면, 이를 보안 격리를 위해 서브네팅하겠습니다. 
외부(인터넷)와 직접 통신해야 하는 웹 서버(프론트엔드)는 라우팅 테이블에 IGW(Internet Gateway)가 연결된 `10.0.1.0/24` 대역의 Public Subnet에 배치하겠습니다. 반면 외부에서 절대 직접 접근하면 안 되는 데이터베이스(DB)나 Safe-Path의 중요 관제 백엔드 서버는 `10.0.2.0/24` 대역의 Private Subnet에 배치하여 보안(기밀성)을 철저히 분리(Isolation)하는 아키텍처를 설계하겠습니다."
