# [Network] 클라우드 아키텍트 관점의 실전 서브넷 및 VPC 설계 (Safe-Path 인프라)

> **작성일:** 2026-07-03
> **작성자:** 최현민 (Infra & Security Architect)
> **학습 주제:** AWS VPC, 3-Tier 아키텍처, 다중 가용 영역(Multi-AZ), 라우팅 테이블

---

## 1. 아키텍트의 시선: 서브네팅을 하는 진짜 이유

단순히 네트워크 이론 시험을 통과하기 위해 IP를 쪼개는 것이 아닙니다. 대기업 클라우드 환경에서 아키텍트가 서브넷을 나누는 진짜 이유는 다음 세 가지입니다.

1. **보안 격리 (Security Isolation):** 외부(인터넷)에서 직접 접속해야 하는 서버와, 절대 외부에서 직접 접근하면 안 되는 DB를 물리적/논리적으로 분리하기 위함입니다.
2. **고가용성 (High Availability):** 데이터센터(AZ) 한 곳에 불이 나거나 정전이 되어도 서비스가 죽지 않도록, 네트워크를 여러 지역(AZ)에 골고루 분산하기 위함입니다.
3. **확장성 (Scalability):** 서비스가 대박이 나서 서버를 100대, 200대로 늘려야 할 때 IP가 모자라지 않도록 넉넉한 여유 공간(Buffer)을 두기 위함입니다.

---

## 2. Safe-Path 클라우드 인프라 (AWS VPC) 설계 시나리오

우리 Safe-Path 프로젝트의 백엔드 서버를 AWS에 구축한다고 가정해 봅시다.

* **회사에 할당된 전체 네트워크 (VPC CIDR):** `10.0.0.0/16` 
  * 32자리 중 16자리가 고정되었으므로, 남은 16자리로 IP를 만듭니다. (사용 가능한 IP: 약 65,000개)
* **목표:** 이 거대한 공간을 쪼개어 **안전한 3-Tier(웹-WAS-DB) 구조**를 만듭니다.

###  [아키텍트의 꿀팁] 왜 하필 10.0.0.0 인가요?
인터넷에서 사용하지 못하도록 국제적으로 약속된 **'사설 IP(Private IP)'** 대역이기 때문입니다. 사설 IP 대역(`10.x.x.x`, `172.16.x.x`, `192.168.x.x`)을 사용해야 클라우드 내부망에서 안전하게 통신할 수 있습니다.

---

## 3. 실전 서브넷 주소 할당 (IP Address Plan)

총 6개의 서브넷으로 쪼개겠습니다. 
재난(데이터센터 화재)을 대비해 가용 영역(AZ)을 A와 C, 두 곳으로 나눕니다. 서브넷 마스크는 모두 **`/24` (IP 256개씩)** 로 통일합니다.

| 용도 (Tier) | 가용 영역 A (AZ-a) | 가용 영역 C (AZ-c) | 특징 및 라우팅 |
| :--- | :--- | :--- | :--- |
| **Public 서브넷**<br>(로드밸런서, 방화벽) | `10.0.10.0/24` | `10.0.11.0/24` | 외부 인터넷과 직접 연결됨 (**IGW** 연결) |
| **Private 서브넷**<br>(백엔드 API 서버) | `10.0.20.0/24` | `10.0.21.0/24` | 외부 접속 불가. 나갈 때만 **NAT Gateway** 이용 |
| **DB 서브넷**<br>(데이터베이스, 캐시) | `10.0.30.0/24` | `10.0.31.0/24` | 외부 접속 100% 차단. 인터넷 접속 아예 불가 |

###  [아키텍트의 꿀팁] 왜 1, 2, 3이 아니라 10, 20, 30으로 건너뛰나요?
초보자들은 서브넷을 `10.0.1.0/24`, `10.0.2.0/24` 처럼 따닥따닥 붙여서 짭니다. 하지만 아키텍트는 **'확장성'**을 고려합니다. 
만약 Public 서브넷의 IP가 모자라서 더 만들어야 한다면? `10.0.12.0/24`, `10.0.13.0/24` 처럼 빈 공간(Buffer)에 손쉽게 추가할 수 있도록 일부러 번호대를 띄워놓는 것입니다. (이것이 실무진의 노하우입니다!)

## 4. 아키텍처의 핵심: 라우팅 테이블(Routing Table) 분리

서브넷을 나눴다고 끝이 아닙니다. 이 네트워크들이 어떻게 통신할지 '길(Route)'을 정해주는 것이 보안의 완성입니다.

1. **Public 서브넷의 길잡이 (Internet Gateway)**
   * 목적지가 내부망(`10.0.0.0/16`)이면 -> 내부로 보냄 (Local)
   * 목적지가 인터넷(`0.0.0.0/0`)이면 -> **인터넷 게이트웨이(IGW)**로 보냄! (외부망 통신 가능)

2. **Private 서브넷의 길잡이 (NAT Gateway)**
   * 목적지가 내부망(`10.0.0.0/16`)이면 -> 내부로 보냄 (Local)
   * 목적지가 인터넷(`0.0.0.0/0`)이면 -> **NAT Gateway**로 보냄! 
   * *(해설: 밖에서 해커가 안으로 들어오는 건 막고, 우리 서버가 업데이트를 위해 잠시 인터넷으로 '나가는 것'만 허용하는 단방향 통로입니다.)*

3. **DB 서브넷의 길잡이 (폐쇄망)**
   * 목적지가 내부망(`10.0.0.0/16`)이면 -> 내부로 보냄 (Local)
   * 인터넷(`0.0.0.0/0`)으로 가는 길 자체를 안 만듦! 
   * *(해설: 부모님의 개인정보가 담긴 DB는 인터넷과 아예 단절시켜 완벽한 제로 트러스트 환경을 만듭니다.)*

## 실전 Q&A

**Q (면접관):** "최현민 지원자님, 백엔드 서버(API)를 그냥 Public Subnet에 두고 보안그룹(포트 차단)만 설정하면 안 되나요? 굳이 Private Subnet을 만들고 NAT Gateway를 쓰면 아키텍처도 복잡해지고 NAT 요금도 나올 텐데요."

**A (나의 답변):**
"좋은 질문이십니다. 물론 보안그룹(Security Group)으로 포트를 막을 수는 있지만, 이는 **심층 방어(Defense in Depth)** 관점에서 부족합니다. 
만약 실수로 백엔드 서버에 공인 IP(Public IP)가 할당되거나 보안그룹 룰을 잘못 설정하는 '휴먼 에러'가 발생하면 서버가 즉시 전 세계의 포트 스캐닝과 해킹 타겟이 됩니다. 

하지만 **Private Subnet에 서버를 배치하면, 논리적으로 인터넷 게이트웨이(IGW)로 향하는 라우팅 자체가 없기 때문에 공인 IP가 부여되더라도 외부에서 접근할 수 있는 물리적/논리적 경로가 원천 차단**됩니다. 즉, 인프라 계층(네트워크)에서 1차로 막고, 인스턴스 계층(보안그룹)에서 2차로 막기 위해 반드시 Private Subnet 분리가 필요하다고 생각합니다. 비용이 들더라도 뚫렸을 때의 리스크(개인정보 유출)가 훨씬 크기 때문입니다."
