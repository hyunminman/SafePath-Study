1. SQL Injection 실습 (Dreamhack: simple_sqli)

가장 기본이 되는 웹 취약점인 SQL Injection을 체화하기 위해 드림핵(Dreamhack) 워게임의 simple_sqli 문제를 풀고 인증 우회를 실습했다.

서버 소스코드 분석 (app.py)
제공된 파이썬 백엔드 코드를 살펴보니 데이터베이스 조회 로직에 치명적인 취약점이 있었다.

res = query_db(f'select * from users where userid="{userid}" and userpassword="{userpassword}"')


사용자의 입력값(userid)을 검증 없이 f-string으로 쿼리에 직접 꽂아 넣고 있었고, 홑따옴표가 아닌 큰따옴표(")를 사용하고 있었다.

공격 페이로드 작성 및 우회 성공
따라서 쿼리 구조를 탈옥하고 뒷부분의 비밀번호 검증 로직을 무력화시키기 위해 다음과 같은 페이로드를 작성해 아이디 칸에 주입했다.

Payload: admin"--  (맨 뒤 공백 포함)

입력 결과, 서버에서 실행되는 실제 쿼리는 다음과 같이 조작되었다.
select * from users where userid="admin"--" and userpassword="..."
뒤의 패스워드 검증 로직이 -- 기호로 인해 주석 처리되면서 무사히 관리자(admin) 계정으로 로그인에 성공했고, 플래그를 획득할 수 있었다. 방어만 외우는 것보다 직접 코드를 보고 뚫어보니 원리가 훨씬 와닿았다.
(※ 워게임 규정에 따라 스포일러 방지를 위해 정답 플래그 DH{...} 원본은 캡처에서 가림 처리함)

2. React XSS 취약점 직접 검증하기

React는 기본적으로 데이터 바인딩 시 문자열 이스케이프(Escape)를 수행하여 XSS 공격을 막아준다. 하지만 dangerouslySetInnerHTML 속성을 잘못 썼을 때 얼마나 위험한지 직접 검증해보고 싶어 CodeSandbox를 통해 실습 환경을 구축했다.

실습 내용
테스트용 텍스트 입력창을 만들고, 악성 스크립트가 담긴 페이로드(<img src="x" onerror="alert('XSS 해킹 성공!')" />)를 입력해 보았다.

결과 분석

안전한 기본 바인딩 ({input}): 입력한 페이로드가 단순한 문자열로 화면에 출력되었다. React가 내부적으로 괄호 < > 등을 안전한 엔티티로 변환해 주었기 때문이다.

취약한 렌더링 (dangerouslySetInnerHTML 사용 시): 코드를 입력하자마자 브라우저 상단에 강제로 Alert 경고창이 팝업되었다.

필터링 없이 브라우저의 DOM에 HTML을 직접 삽입하게 되면 스크립트가 그대로 실행됨을 눈으로 확인했다. 향후 프로젝트에서 에디터 입력값 등을 렌더링해야 할 때는 반드시 DOMPurify와 같은 라이브러리로 사전 무해화(Sanitization) 작업이 필수적임을 깨달았다.

3. L4/L7 로드밸런서 아키텍처 학습

그동안 텍스트로만 읽었던 네트워크 분산 처리 과정을 유튜브 아키텍처 강의(우아한형제들 로드밸런서 등)를 시청하며 다이어그램으로 머릿속에 시각화했다.

L4 로드밸런서 (NLB): IP 주소와 포트(TCP) 정보만 보고 패킷을 빠르게 분산시킨다. 단순 무식하지만 처리 속도가 엄청나게 빠르다는 장점이 있다.

L7 로드밸런서 (ALB): HTTP 패킷의 내부(헤더, URL Path 등)를 열어보고 똑똑하게 라우팅한다. 특히 이 과정에서 WAF(웹 방화벽)와 연동하여 SQLi나 XSS 같은 악의적인 페이로드를 사전에 필터링할 수 있다는 점이 보안 관점에서 매우 중요하다는 것을 배웠다.

느낀 점
애플리케이션(코드) 단에서의 시큐어 코딩뿐만 아니라, L7 로드밸런서와 같은 인프라 단에서의 방어가 결합될 때 비로소 견고한 보안 아키텍처가 완성됨을 이해할 수 있었다.
